The SilkNotes Promise

Knowledge lives in silos. Your thoughts are scattered across Obsidian vaults, Logseq graphs, Notion workspaces, Tana nodes, and a dozen other tools — each with its own format, its own lock-in, its own partial view of what you know. SilkNotes exists to dissolve these walls.

The promise rests on three pillars:

  • Comprehensive feature set — everything you need in a knowledge operating system, from backlinks and canvas views to structured output.
  • Interoperate with any data silo — your notes flow freely between tools. No exports, no copy-paste, no manual reconciliation.
  • Silky in use — effortless and smooth. The complexity of interoperation stays invisible; the experience stays fluid.

As a sovereign basis, SilkNotes will host its own local-first note-operating system with an open-source free tier and premium support — ensuring that your knowledge always belongs to you.

Current State

SilkNotes is in active early development. The closest existing approximation to the vision is Logseq with its whiteboard feature — but the interoperability layer that SilkNotes aims to build goes far beyond any single tool.

Roadmap

Development follows a practical sequence: prove interoperation between existing tools first, then build inward toward the sovereign core.

  1. Get Obsidian to work — establish the foundational file-level bridge and prove bidirectional sync.
  2. Interoperate with Logseq — extend the Logseq Plugin sub-holon to handle graph-level data exchange.
  3. Interoperate with other FOSS tools — generalize the bridge architecture to Reor, Affine, Legend App, and beyond.
  4. Recreate core features — build SilkNotes' own editing surface with backlinks, canvas, and structured output.
  5. SilkNotes full on — launch the sovereign, local-first platform with open-source free tier and premium support.

Interoperability Approach

SilkNotes treats interoperability as a layered problem. Rather than building one monolithic converter, it creates typed bridges between tool categories.

FOSS Internal Bridges

Tools that share open formats can be connected deeply. Logseq, Obsidian, Cursor, and other open-source tools communicate through file-system-level sync and shared data structures. This is the highest-fidelity layer.

FOSS-to-Proprietary Bridges

Tools like Notion and Todoist expose APIs but control the format. SilkNotes builds adapter layers that translate between proprietary schemas and its own open interchange format — preserving as much structure as the API allows.

The Output API

Every bridge feeds into a unified output API, so downstream consumers (dashboards, publishing pipelines, AI agents) can access your knowledge regardless of which tool originated it.

Tech Vision

The long-term architecture centers on conflict-free collaboration and sovereignty:

  • CRDT data structure — conflict-free replicated data types ensure that edits from any device or tool merge cleanly without a central authority.
  • WebSocket sync — real-time propagation of changes across connected clients and bridges.
  • PostgreSQL persistence — durable storage layer for the canonical state, queryable and portable.
  • Local-first by default — your data lives on your machine. The server is optional, never required.
  • Open source core — the free tier includes the full editing surface, all FOSS bridges, and local storage. Premium support covers hosted sync, priority bridges, and enterprise features.

Feature Wish List

  • Easy analogue backlinks — linking should feel as natural as writing
  • Canvas / whiteboard view
  • Interoperate with external data sources beyond note-taking tools
  • Output API for downstream consumers
  • Fully offline-capable with seamless reconnection

AI-powered private note-taking. Investigating bi-directional sync with SilkNotes' CRDT layer.

Network thinking and text analysis. Exploring how graph-based insight tools can plug into the SilkNotes interchange format.
